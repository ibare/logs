에이전트의 3대 체계적 약점

  • 파일 크기/구조 위반 (26%) — 렌더러 파일이 300줄 제한을 초과하거나 여러 도메인 로직이 한 파일에 혼합. 에이전트는 기능 구현에 집중하면서 파일 크기 제약을 잊는 경향이 있다.
  • import/경로 정합성 위반 (20%) — 리네이밍 시 영향받는 모든 파일을 파악하지 못함. 에이전트의 '영향 범위 추론' 능력이 가장 취약한 영역.
  • 타입 안전성 위반 (17%) — any 타입 사용, 타입 가드 누락. 빠르게 코드를 만들려는 압박이 타입 정밀도를 희생시킨다.

핵심 발견: 규칙 복잡도와 위반율은 반비례한다

직관과 반대로, 아이템이 많은 규칙(S-konva 45개, S-server 30개)이 오히려 위반율 0%를 기록한다. 반면 아이템이 적은 규칙(S-compiler 20개)이 최고 위반율 8.24%를 보인다.

  • 상세한 규칙은 에이전트를 더 잘 가이드한다 — 구체적 예시, DO/DON'T 패턴이 많을수록 에이전트가 명확하게 따른다.
  • 위반은 규칙의 부족이 아니라 도메인의 순서 의존성에서 발생한다 — S-compiler의 '패스 순서' 같은 전역적 제약은 아이템 수와 관계없이 위반하기 쉽다.
  • 독립적 규칙 vs 순서 의존 규칙 — S-konva(각 컴포넌트가 독립)는 하나를 지키면 다른 것에 영향 없다. S-compiler(패스 A → B → C 순서)는 하나를 바꾸면 전체가 흔들린다.
09

결론: 규칙 시스템 설계를 위한 교훈

01

상세함이 준수를 만든다

45개 아이템 규칙(위반 0%)이 20개 아이템 규칙(위반 8%)보다 낫다. 구체적 예시와 코드 패턴이 에이전트를 올바른 길로 이끈다.

02

순서 의존 규칙에 특별히 주의하라

파이프라인, 컴파일 패스, 마이그레이션 순서 같은 규칙은 위반 확률이 높다. 이런 규칙에는 자동 검증 메커니즘이 필요하다.

03

2계층 구조가 작동한다

공유 Concerns(위반율 2.3%)가 도메인 Specifics(위반율 4.4%)보다 잘 지켜진다. 범용 원칙을 먼저 확립하고, 도메인 규칙을 점진적으로 추가하는 전략이 유효하다.

04

에이전트의 맹점은 '영향 범위 추론'

위반의 46%가 '다른 파일/문서/규칙과의 정합성 누락'에서 발생. 에이전트는 현재 파일에 집중하면서 파급 효과를 과소평가한다.
